# Next.js Middleware Bypass (CVE-2025-29927) — JavaScript/Next.js

## 📅 When Did It Happen?
March 2025
Affected versions: Next.js 11.1.4 - 15.2.2

## 🎯 Target System
Next.js — the most popular React framework. ~15% of websites use Next.js. The middleware mechanism was targeted.

## 🔴 What Happened?
CVE-2025-29927 is a critical **authentication bypass** vulnerability in Next.js's middleware mechanism.

**How it worked:**
Next.js middleware is a security layer that runs before pages are loaded. Middleware:
- Checks user authentication
- Applies geographic restrictions
- Adds CSP headers
- Performs redirects

However, Next.js used a **depth counter** to prevent middleware from entering infinite loops. This counter was stored in the `x-middleware-subrequest` HTTP header. **Critical flaw: this header could be controlled by the client.**

The attacker only needed to:
```
x-middleware-subrequest: middleware:middleware:middleware:middleware:middleware
```
Adding this header to the request told Next.js that middleware had already run 5 times and should be skipped. Result: all security checks were bypassed.

## 🧠 Root Cause
1. **Trusting an HTTP header** — Next.js used a client-sent header as a middleware counter
2. **Recursion depth control** — the mechanism designed to prevent infinite loops became a security vulnerability
3. **Client-controllable header** — the `x-middleware-subrequest` header was not validated, removed, or rewritten

## 💥 Impact
- **Full authentication bypass** — attacker could access admin panels, private dashboards, user data
- **CSP bypass** — Content Security Policy headers could be skipped, enabling XSS attacks
- **Geographic restriction bypass** — access to region-restricted content
- All Next.js versions released for years (11.1.4 to 15.2.2) were affected
- Millions of websites were at risk

## 🔧 How Was It Fixed?
- Patched in Next.js 15.2.3+ and 14.2.25+
- The `x-middleware-subrequest` header can no longer be controlled by the client
- Workaround: block the header in NGINX/Apache
- Vercel published a security advisory

## 🎓 Lessons Learned
1. **Never trust HTTP headers** — everything sent by the client can be manipulated
2. **Infinite loop protection must not create security vulnerabilities** — such mechanisms should be managed with a separate counter (on the server side)
3. **Defense in depth** — middleware should not be your only security layer; also perform validation on the backend
4. **Keep framework versions up to date** — a simple bug may go unnoticed for years
5. **The simplest bugs are the most dangerous** — a single HTTP header could bypass all security

## Vibe Coding Connection: How Can This Error Recur in AI Code Generation?
When you ask AI to "write a Next.js middleware," AI generally produces correct and secure middleware. However, AI cannot know about security vulnerabilities in the framework itself. When you ask AI to "create a Next.js project," AI might use an old Next.js version (even though `npx create-next-app` installs the latest version, AI sometimes specifies a fixed version). More importantly, code produced by AI might use an architecture that relies on middleware and forget to add a second validation layer in the backend. This leads to the misconception in AI code that "middleware is secure, backend validation is unnecessary." In Next.js projects generated by AI, always make sure there are authorization checks on the backend side as well.
