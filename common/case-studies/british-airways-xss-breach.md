# British Airways / Magecart XSS Attack — XSS / JavaScript

## 📅 When Did It Happen?
2018 (August-September), discovery: September 2018

## 🎯 Target System
British Airways website (ba.com) — the site loaded its own copy of the **Modernizr** JavaScript library (an old build, v2.6.2).

## 🔴 What Happened?
The Magecart hacker group (famous for credit card skimming attacks) compromised British Airways' own website scripts by **modifying the site's Modernizr JavaScript file** and injecting a payment-skimming routine.

Attack flow:
1. Attackers gained access and **modified British Airways' own Modernizr script** (v2.6.2), appending ~22 lines of malicious JavaScript
2. The injected code serialized the payment form contents and added a skimmer to the payment page
3. The script stole customers' payment information (credit card number, CVV, expiry date)
4. The stolen data was sent to a fake server with a domain name resembling British Airways' real domain (`baways.com`)
5. A fake certificate (SSL) was used, so users thought they were on a secure connection

## 🧠 Root Cause
1. **Modification of a first-party script (Modernizr)** — the site's own JavaScript was altered without detection
2. **Insufficient script security** — the website could not detect injected/modified scripts
3. **Lack of CSP (Content Security Policy)** — a strict enough CSP policy could have prevented script injection
4. **Lack of Subresource Integrity (SRI)** — the library was loaded without integrity checks

## 💥 Impact
- **380,000** payment transactions compromised
- British Airways fined **£20 million** by the ICO (GDPR violation)
- One of the first major GDPR fines (initially proposed at £183 million, reduced due to the pandemic)
- Customer trust severely damaged
- Brand reputation damage

## 🔧 How Was It Fixed?
- The tampered Modernizr script was removed/restored on the website
- Security vulnerability patched
- CSP (Content Security Policy) implemented
- Subresource Integrity checks added
- All third-party JavaScript dependencies audited
- IT infrastructure strengthened

## 🎓 Lessons Learned
1. **Third-party scripts are the weakest link** — each one is a potential attack vector.
2. **Use CSP (Content Security Policy)** — whitelist which scripts can execute.
3. **Subresource Integrity (SRI)** — check integrity hash for every external script.
4. **Magecart-type attacks** — script injections that steal form data are one of the biggest threats to e-commerce.
5. **GDPR fines are severe** — £20 million demonstrates the cost of not investing in security measures.

## Vibe Coding Connection: How Can This Error Be Repeated in AI Code Generation?
When creating a web page with AI, the AI easily adds third-party JavaScript code with requests like "add analytics, add push notification, include this popular library." AI does not assess the security of these libraries and does not add CSP/SRI. An e-commerce page generated by AI could unknowingly share customer data with third parties. Audit all external scripts and add CSP to HTML templates generated by AI.
