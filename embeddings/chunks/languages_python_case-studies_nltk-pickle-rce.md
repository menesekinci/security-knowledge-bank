# NLTK Pickle Deserialization RCE (CVE-2024-39705) — Python

## 📅 When Did It Happen?
Discovery: 2024
CVE published: CVE-2024-39705

## 🎯 Target System
NLTK (Natural Language Toolkit) — a popular natural language processing library for Python. It had a security vulnerability when loading model files in Pickle format.

## 🔴 What Happened?
CVE-2024-39705 is a critical security vulnerability that arises when **deserializing untrusted data** using the Python Pickle library.

Pickle is a built-in library used in Python for serializing and deserializing objects. The fundamental problem with Pickle: **any deserialized pickle file can execute arbitrary Python code**. The Pickle format allows executing any callable object during deserialization via the `__reduce__` method.

This vulnerability emerged because NLTK loads models in Pickle format — if a model file comes from an untrusted source (download, user upload, etc.), the malicious pickle payload inside it can lead to RCE.

## 🧠 Root Cause
1. **Inherently unsafe deserialization of Pickle**: Pickle was not designed for untrusted data
2. **Loading untrusted pickle files**: NLTK was loading model files from external sources using pickle
3. **Lack of input validation**: file content was not validated before calling pickle.load()
4. **Alternative formats not used**: pickle was preferred over JSON, MessagePack, or safe alternatives

## 💥 Impact
- **CVSS 9.8** (Critical) — remote code execution
- An attacker can execute arbitrary code on the server by sending a specially crafted pickle file
- In ML pipelines, this means compromising the entire model training/serving infrastructure
- Model poisoning and data leakage are possible

## 🔧 How Was It Fixed?
- NLTK security patch released
- Pickle loading points were secured
- Alternative safe serialization formats (like safetensors) were recommended
- Users were advised not to load untrusted pickle files

## 🎓 Lessons Learned
1. **Pickle is not secure** — a call to `pickle.load(data)` can execute any code, regardless of its contents
2. **NEVER load pickle files from untrusted sources**
3. **Use alternative formats**: `safetensors` (for ML models), `JSON`, `MessagePack`, `PyArrow`
4. **Digital signature verification** — sign pickle files and verify them when loading
5. **Sandboxing** — if pickle loading is necessary, do it in an isolated environment

## Vibe Coding Connection: How this error can be reproduced in AI code generation
AI is highly prone to generating code that saves and loads ML models using pickle. `torch.save(model, 'model.pth')` and `torch.load(model)` (which uses pickle) are frequently generated by AI. When you ask AI to "train and save a model", it's almost guaranteed to use pickle-based serialization. When you see `pickle.load()` or equivalents (`torch.load()`, `joblib.load()`) in AI-generated ML code, replace them with a safe alternative like `safetensors`.
