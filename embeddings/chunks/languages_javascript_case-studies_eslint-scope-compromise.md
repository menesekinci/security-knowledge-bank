# eslint-scope npm Account Takeover Attack — JavaScript/Node.js

## 📅 When Did It Happen?
July 12, 2018

## 🎯 Target System
ESLint — the most popular static code analysis tool for JavaScript. The eslint-scope and eslint-config-eslint packages were targeted.

## 🔴 What Happened?
An attacker **compromised an ESLint maintainer's npm account** and published malicious package versions.

Timeline:
1. **Before the attack**: The attacker found the maintainer's email and password in a third-party data breach (from another website)
2. **July 12, 2018, early hours**: The attacker created an authentication token on the maintainer's npm account
3. **09:49 UTC**: `eslint-config-eslint@5.0.2` was published — with a malicious postinstall script
4. **10:25 UTC**: eslint-config-eslint was removed (experimental)
5. **10:40 UTC**: `eslint-scope@3.7.2` was published — with the same malicious script
6. **11:17 UTC**: A user reported the issue on GitHub
7. **12:27 UTC**: The malicious pastebin link was removed
8. **12:37 UTC**: The npm team removed eslint-scope@3.7.2
9. **18:42 UTC**: npm revoked all tokens created before 2018-07-12 12:30 UTC

**What was the malicious code doing?**
- The `postinstall` script ran during npm install
- It downloaded and executed code from `pastebin.com`
- This code sent the npm authentication token from the `.npmrc` file to the attacker
- The attacker's goal was to steal tokens to compromise more npm packages

## 🧠 Root Cause
1. **Password reuse** — the maintainer used the npm password on other sites as well
2. **Lack of two-factor authentication (2FA)** — the maintainer's npm account had no 2FA
3. **npm account security** — without 2FA, a password leak leads to account takeover
4. **postinstall scripts** — npm allows arbitrary script execution during package installation

## 💥 Impact
- eslint-scope was a dependency of many popular packages (eslint, babel-eslint, webpack)
- Thousands of developers' npm tokens were at risk
- npm forcibly revoked all old tokens (a first in npm history)
- Trust was shaken across the entire npm ecosystem
- The ESLint team completely revamped their security processes

## 🔧 How Was It Fixed?
- Malicious packages were removed from npm
- npm forcibly revoked all old tokens
- ESLint published eslint-scope@3.7.3 (clean)
- ESLint made 2FA mandatory for all maintainers
- npm increased 2FA requirements and account security
- ESLint published a comprehensive postmortem

## 🎓 Lessons Learned
1. **2FA must be mandatory for everyone** — especially package maintainers
2. **Use a password manager** — never reuse passwords
3. **npm postinstall scripts are dangerous** — these scripts can execute arbitrary code
4. **Use lockfiles** (`package-lock.json` / `yarn.lock`) — version pinning prevents automatic malicious updates
5. **Incident response plan** — ESLint's rapid response and transparent postmortem should be emulated

## Vibe Coding Connection: How Can This Bug Be Repeated in AI Code Generation?
Projects generated by AI often add tools like ESLint/Prettier as development dependencies. When AI recommends `npm install eslint --save-dev`, it does not check for safe versions. Additionally, if AI-generated CI/CD pipelines include npm publish operations, AI may not add security checks (2FA, token rotation). Add audit steps before `npm publish` in AI-generated npm configurations.
