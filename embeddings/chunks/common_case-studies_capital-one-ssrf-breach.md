# Capital One Data Breach — SSRF / AWS Metadata (Web/Cloud)

## 📅 When Did It Happen?
March 22-23, 2019 (attack), July 19, 2019 (discovery), July 29, 2019 (public disclosure)

## 🎯 Target System
Capital One — one of the largest banks in the US. It was running on AWS cloud infrastructure. It held data for 106 million credit card applicants.

## 🔴 What Happened?
Paige Thompson, a former AWS software engineer, exploited an **SSRF (Server-Side Request Forgery)** vulnerability in Capital One's cloud infrastructure.

Attack chain:
1. Capital One's **Web Application Firewall (WAF)** was misconfigured
2. The attacker used an SSRF vulnerability in an application behind the WAF to send a request to `http://169.254.169.254/latest/meta-data/` (AWS EC2 Instance Metadata Service)
3. The metadata service returned the **temporary credentials** (AccessKeyId, SecretAccessKey, SessionToken) of the IAM role assigned to the EC2 instance
4. Using these credentials, the attacker accessed Capital One's S3 buckets and downloaded 30 GB of data

## 🧠 Root Cause
1. **SSRF vulnerability**: The application fetched URLs based on user input without validation.
2. **Overly permissive IAM role**: The WAF instance was assigned an IAM role with more permissions than necessary.
3. **Use of IMDSv1**: IMDSv1 was used instead of IMDSv2 (token-based), which made SSRF easier.
4. **WAF misconfiguration**: The firewall was configured in a way that could not block the SSRF attack.

## 💥 Impact
- **106 million** people's data leaked (US and Canada)
- **140,000** Social Security numbers
- **80,000** linked bank account numbers
- **1 million** Canadian Social Insurance Numbers (SIN)
- **~$300 million** in financial losses (including lawsuits and settlements)
- **$80 million** OCC (Office of the Comptroller of the Currency) fine
- The bank's reputation was severely damaged

## 🔧 How Was It Fixed?
- IAM roles were narrowed down (least privilege principle)
- Migrated to IMDSv2
- WAF rules were strengthened
- Additional encryption and automated compliance audits were added
- Multi-factor authentication (MFA) was enforced across all systems
- Paige Thompson was convicted in 2022

## 🎓 Lessons Learned
1. **SSRF is much more dangerous in cloud environments** — it can provide access to metadata services.
2. **The principle of Least Privilege is vital.**
3. **Use IMDSv2** — completely disable IMDSv1.
4. **WAF must be properly configured and audited regularly.**
5. **Cloud security is a shared responsibility** — AWS protects the infrastructure, you protect your applications.

## Vibe Coding Connection: How this error can be reproduced in AI code generation
When generating code with AI, a prompt like "fetch content from a user-supplied URL" can easily result in a function vulnerable to SSRF. Direct calls like `requests.get(user_input)` or `fetch(user_input)` are particularly common. If you ask AI to "create a proxy endpoint that pulls data from an external source", AI will likely generate code without any URL validation. Audit all HTTP client code generated by AI for URL validation and allowlist checking.
