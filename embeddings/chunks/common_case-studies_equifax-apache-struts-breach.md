# Equifax Data Breach — Apache Struts (Java)

## 📅 When Did It Happen?
May-July 2017 (discovery: July 29, 2017, public disclosure: September 7, 2017)

## 🎯 Target System
Equifax — One of the three largest credit bureaus in the US. Held sensitive data on 143 million American consumers. Used Apache Struts, a Java-based web application framework.

## 🔴 What Happened?
Attackers exploited the Apache Struts CVE-2017-5638 vulnerability in Equifax's customer dispute portal. This vulnerability was a **Remote Code Execution (RCE)** flaw in the Struts Jakarta Multipart parser's "Content-Type" header. The attacker could execute arbitrary Java code on the server by sending a specially crafted HTTP Content-Type header.

The vulnerability was published **March 7, 2017** — Equifax had 2 months to patch it. The attack occurred **May 13, 2017**.

## 🧠 Root Cause
1. **Patch failure**: Equifax did not apply the patch for CVE-2017-5638 within the 2-month window. Patching was laborious — it involved downloading an updated version of Struts and then using it to rebuild all applications using the old, vulnerable Struts versions.
2. **Inventory blindness**: Equifax did not have a complete inventory of all applications using Struts.
3. **Lack of patch management**: No robust process for tracking and patching security vulnerabilities.

## 💥 Impact
- **143 million** people's personal data leaked (SSNs, dates of birth, addresses)
- **209,000** credit card numbers stolen
- **$200 million** direct remediation cost
- Stock fell 35%
- CEO, CSO, and CIO retired / were fired
- **$700 million** total settlements and fines (FTC, CFPB, states)
- Congressional investigations and lawsuits from 50 states

## 🔧 How Was It Fixed?
- Apache Struts patch applied (though belatedly)
- Patch management processes overhauled
- Network segmentation improved
- Web application firewall added
- All affected systems shut down for forensic imaging
- Credit monitoring and identity protection provided for affected consumers

## 🎓 Lessons Learned
1. **Apply patches without delay** — SLAs for critical vulnerabilities should be hours, not months.
2. **Maintain an SBOM (Software Bill of Materials)** — if you don't know which libraries are used where, you can't patch them.
3. **Defense in depth** — a single framework vulnerability should not bring down the entire system.
4. **Inventory management is critical** — you cannot protect an asset you don't know exists.

## Vibe Coding Connection: How Can This Error Be Repeated in AI Code Generation?
When generating code with AI, relying on AI to update dependencies is dangerous. A project generated by AI may include an unpatched Apache Struts (or similar) dependency. Equifax's mistake was **human neglect of the patch process** — AI could worsen this neglect because developers may assume "AI produced secure code." Always manually audit `pom.xml` or `build.gradle` files generated by AI.
